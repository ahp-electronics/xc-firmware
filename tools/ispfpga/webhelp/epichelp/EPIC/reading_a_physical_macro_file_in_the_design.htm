Reading a Physical Macro File in the Design

A schematic or hardware description language (HDL) source file might contain a reference to a physical macro library file. The macro library file, which has an .nmc extension, can be included as a block in the design's schematic. The reference appears as a FILE= filename property attached to the block representing the physical macro, with the filename pointing to the .nmc macro library file.

If you include a physical macro in your schematic or HDL, ispLEVER or Diamond processes the macro in the following way:

1. The netlist reader passes the macro to the .ngo file as an unexpanded block. No information about the block's internal logic and connectivity is transferred, only the applicable .nmc name.
2. The Build Database process, Translate Design process, or NGDBUILD finds the macro library file and resolves the macro's logic and connectivity to the design primitives. This information becomes part of the .ngd file produced. Since the .ngd contains only a logical representation of the design, no relative placement or routing information for the macro appears in the file. NGDBUILD also places the full path name of the .nmc into the file.
Note
The names of the block's input and output pins in the schematic must match the names of the external pins specified in the physical macro's library file (the .nmc file that defines the physical macro). If pin names do not match, this error message will be generated:
Pin mismatch between block
instance_name, FILE= filename and file
filename.nmc at pin pin_name
pin_name identifies a pin found on the schematic symbol but not in the .nmc file.
